# BPM Concepts Reference

A condensed reference of all BPM concepts discussed during design.
Use this as a quick lookup when implementing engine features.

---

## Core Concepts

### Process Definition
The blueprint — a BPMN XML file describing the flow, tasks, gateways, and events.
Deployed once, instantiated many times. Versioned.

### Process Instance
A running execution of a process definition. Like an object instantiated from a class.
Each instance has a unique ID and its own variables.

### Token (Execution)
A pointer that tracks where in the flow the process currently is.
In reality: an EXECUTION row in the DB with an `activity_id`.
Multiple tokens can exist in one instance (parallel paths).

### Process Variables
The working memory of a process instance. Key-value pairs.
Set at start, readable/updatable by any step throughout the lifecycle.

### Wait State
A point where the token pauses and the engine thread is freed.
All wait state is persisted to DB. Engine is stateless between steps.

## Task Types

| Task | Automated | Waits | Runs In | Use Case |
|---|---|---|---|---|
| UserTask | ❌ | ✅ | Human via UI | Approval, review |
| ServiceTask | ✅ | Sometimes | External worker | Call API, system |
| ScriptTask | ✅ | ❌ | Inside engine | Variable transformation |
| SendTask | ✅ | ❌ | External | Emit a message |
| ReceiveTask | ✅ | ✅ | Engine waits | Wait for message |
| ManualTask | ❌ | ✅ | Off-system | Physical work |
| BusinessRuleTask | ✅ | ❌ | DMN engine | Decision table |
| CallActivity | ✅ | Depends | Subprocess | Reuse another process |

### Script Task vs Service Task
```
Script Task  → runs INSIDE the engine (inline computation only)
Service Task → runs OUTSIDE the engine (external worker or connector)

Rule: if it touches anything outside process variables → Service Task
      if it could block or fail in ways that risk the engine → Service Task
      if it's a one-liner variable transform → Script Task
```

## Event Types

### Carriers (what transports events)
```
Message → targeted, one-to-one, needs correlation key
Signal  → broadcast, one-to-many, no correlation
Timer   → self-generated by engine (not really a carrier)
```

### Event Positions
```
Start Event         → catches trigger, creates instance
Intermediate Catch  → pauses flow, waits for trigger
Intermediate Throw  → emits trigger, continues immediately
Boundary Event      → watches task while it runs
End Event           → emits on process completion
```

### Boundary Event Variants
```
Interrupting (solid border)    → cancels task, follows boundary flow
Non-Interrupting (dashed border) → spawns parallel path, task continues
```

## Gateway Types

### Exclusive Gateway (XOR ✕)
```
One path in → evaluate conditions top-to-bottom → exactly one path out
Merge: first arrival passes through immediately (no waiting)
Use: if/else routing
```

### Parallel Gateway (AND +)
```
One path in → all outgoing paths fire → multiple tokens out
Merge: waits for ALL paths to complete → one token out
Use: fork and join, parallel execution
```

### Inclusive Gateway (OR ○)
```
One path in → evaluate ALL conditions → one or more paths out
Merge: waits for exactly the paths that were activated
Use: optional multi-path, discount rules, conditional parallelism
```

### Event Based Gateway (◇)
```
Process PAUSES → watches multiple events → first event wins → that path taken
Others discarded
Use: race between events, payment success vs failure vs timeout
```

### Complex Gateway (*)
```
Custom condition defines both split and merge behaviour
Use: quorum/voting (2 of 3 must approve), last resort
```

### Gateway Matching Rule
```
Parallel split   →  Parallel merge   (mismatch = deadlock)
Exclusive split  →  Exclusive merge  (mismatch = duplicate tokens)
Inclusive split  →  Inclusive merge  (mismatch = incorrect wait)
```

## Token Mechanics

### What a Token IS in the DB
```sql
EXECUTION row:
  id                  UUID
  process_instance_id UUID
  activity_id         VARCHAR   -- where the token IS
  state               VARCHAR   -- ACTIVE, WAITING, COMPLETED
  is_concurrent       BOOLEAN   -- in a parallel branch?
  parent_id           UUID      -- for subprocesses
```

### Token at Each Element Type

| Element | Token Behaviour |
|---|---|
| Start Event | Created here |
| End Event | Consumed, instance completed if last |
| UserTask | Paused, task record created |
| ServiceTask | Paused, external task job created |
| ScriptTask | Executes inline, immediately advances |
| Exclusive GW | Passes through, one path |
| Parallel Split | Multiplied — one per outgoing flow |
| Parallel Merge | All must arrive → consumed → one new token |
| Intermediate Catch | Paused, subscription created |
| Intermediate Throw | Passes through immediately |
| Subprocess | Parent paused, child token created inside |

## Wait State Mechanisms

All wait states follow the same pattern:
```
1. Persist wait to DB (task row, job row, subscription row)
2. Free the engine thread
3. External trigger arrives (API call, timer, event)
4. Engine loads state from DB and resumes
```

| Mechanism | Stored As | Woken By |
|---|---|---|
| User/Manual Task | TASK row | Human completes via API |
| Message Correlation | EVENT_SUBSCRIPTION row | POST /messages |
| Signal | EVENT_SUBSCRIPTION row | POST /signals |
| Timer | JOB row with due_date | Job executor poll |
| External Task | JOB row with topic | Worker fetch-and-lock |
| Async Continuation | JOB row | Job executor immediately |

## Off-System (Manual) Tasks

```
Engine reaches manual step
        ↓
Emits notification (email, ticket, API call)
        ↓
Human does physical work
        ↓
Human signals back (link click, ticket resolve, API call)
        ↓
Engine resumes
```

Always add a timer boundary event to prevent hanging indefinitely.

## BPMN Structural Elements

### Pool
Represents a participant (organisation or system).
- Sequence flow CANNOT cross pool boundaries
- Pools communicate ONLY via message flows (dashed arrows)
- Each pool has its own independent process

### Lane
Subdivides a pool into roles or departments.
- Sequence flow CAN cross lane boundaries freely
- Purely visual/organisational — no runtime impact
- Crossing a lane boundary = handoff between roles

### Subprocess Types
```
Embedded        → part of parent, shares variables, grouped tasks
Call Activity   → references separate process definition, reusable
Event Subprocess → triggered by event, not sequence flow
Transaction     → ACID-like, compensation on failure
```

## Engine vs Worker

```
Engine  →  knows WHAT to do and WHEN (orchestrator)
           reads BPMN, moves tokens, manages state
           knows nothing about business logic

Worker  →  knows HOW to do it (executor)
           does the actual work, calls external systems
           knows nothing about the process flow
```

### Worker Communication Pattern (preferred: Pull)
```
Worker polls engine: "any tasks for topic X?"
Engine responds: "yes, here is one, locked for you"
Worker does work
Worker reports back: "done, here are results"
Engine advances token
```

## DMN (Decision Model and Notation)

Complement to BPMN — models the WHAT of a decision, not the HOW of the flow.

### Decision Table
```
Inputs (columns) → Rules (rows) → Outputs (columns)

| Credit Score | Income  | → | Decision | Rate |
|---|---|---|---|---|
| >= 750       | >= 50000 | | Approved | 3.5% |
| >= 700       | >= 40000 | | Approved | 4.5% |
| < 600        | -        | | Rejected | -    |
```

### Hit Policies
```
UNIQUE      → exactly one rule matches (error if multiple)
FIRST       → first matching rule wins (top to bottom)
ANY         → multiple can match but must produce same output
COLLECT     → all matching rules collected into list
COLLECT SUM → sum of all matching numeric outputs
```

### When to Use DMN vs Code
```
DMN:  multi-condition decisions, frequently changing rules,
      business-owned logic, naturally tabular structure

Code: external system calls, complex algorithms, stateful logic,
      looping/iteration, transaction management
```

## Message Structure

```json
// Message (targeted)
{
  "messageName": "PaymentReceived",
  "correlationKey": "ORD-123",
  "variables": { "amount": 250, "currency": "USD" }
}

// Signal (broadcast)
{
  "name": "SystemMaintenanceComplete",
  "variables": { "reason": "scheduled" }
}
```

Key difference: correlation key routes to exactly one instance.
Without it, all listening instances receive it (signal behaviour).

## FEEL Expression Examples (Gateway Conditions)

```
// Comparison
${amount > 1000}
${creditScore >= 700}

// Equality
${plan == "premium"}
${status != "rejected"}

// Compound
${creditScore >= 700 && income > 40000}

// Range
${amount >= 100 && amount < 500}

// String
${contains(email, "@")}
${startsWith(status, "APP")}

// Null check
${approvalResult != null}

// Boolean
${isApproved}
${not(isBlacklisted)}
```
